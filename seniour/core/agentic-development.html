Що таке agentic development і як застосовувати його в iOS-розробці?

Коротка відповідь

Agentic development — це підхід, у якому розробник працює разом з AI-агентом: формулює ціль, дає контекст, просить агента дослідити код, запропонувати або внести зміни, запустити перевірки й пояснити результат. Це не заміна інженерного мислення, а спосіб швидше пройти цикл: зрозуміти задачу, змінити код, перевірити, відрефакторити й задокументувати рішення.

Для Senior iOS Developer важливо вміти не просто “попросити AI написати код”, а керувати роботою агента: обмежувати scope, вимагати тести, перевіряти безпеку, помічати hallucinations, рев'ювити diff і не втрачати ownership над архітектурними рішеннями.

Навчальний розбір

Що треба зрозуміти

Agentic development працює найкраще, коли агент отримує конкретну мету, доступ до релевантного коду, критерії готовності та дозвіл діяти маленькими кроками. Розробник лишається відповідальним за рішення, а агент бере на себе дослідження, рутинні зміни, генерацію тестів, пошук регресій і підготовку пояснення.

Як пояснювати на співбесіді

Говори не про “магію AI”, а про engineering workflow: context gathering, plan, implementation, verification, review. Хороша відповідь показує, що ти вмієш прискорювати delivery без втрати якості, privacy, security і командних стандартів.

Типові помилки

  • Давати агенту широку задачу без критеріїв готовності й меж змін.
  • Приймати згенерований код без рев'ю, тестів і запуску застосунку.
  • Передавати приватні ключі, tokens, PII або внутрішні дані без політик безпеки.
  • Просити агента переписати архітектуру там, де потрібна локальна зміна.

Практика

Візьми невелику iOS feature, наприклад error state для екрана профілю. Попроси агента знайти відповідний ViewModel, додати стан помилки, оновити UI, написати unit tests і пояснити diff. Після цього самостійно перевір, чи не зламана навігація, локалізація, accessibility і loading flow.

Приклад робочого промпта

Goal: Add an empty state to ProfileView when the user has no posts.
Context: This is an iOS app using SwiftUI + MVVM. Keep changes scoped to Profile feature.
Tasks:
1. Inspect ProfileView, ProfileViewModel, and existing tests.
2. Propose the smallest implementation plan.
3. Add the empty state with localized copy and accessibility label.
4. Add or update tests for loading, content, empty, and error states.
5. Run the relevant test command and summarize the diff.

Такий промпт задає ціль, контекст, межі й очікувану перевірку. Він зменшує шанс, що агент піде переписувати unrelated код або пропустить важливі production-вимоги.

iOS-приклад: агент допомагає з тестованістю

protocol ProfileLoading {
    func loadProfile() async throws -> Profile
}

@MainActor
final class ProfileViewModel: ObservableObject {
    enum State {
        case idle
        case loading
        case loaded(Profile)
        case failed(String)
    }

    @Published private(set) var state: State = .idle
    private let loader: ProfileLoading

    init(loader: ProfileLoading) {
        self.loader = loader
    }

    func load() async {
        state = .loading
        do {
            state = .loaded(try await loader.loadProfile())
        } catch {
            state = .failed("Не вдалося завантажити профіль")
        }
    }
}

У цьому прикладі агент може швидко запропонувати protocol abstraction для тестів, fake loader і сценарії перевірки. Завдання Senior-розробника — оцінити, чи цей abstraction справді потрібен у feature, чи він лише додає зайвий boilerplate.

Чекліст для production-використання

  • Перед стартом: сформулюй goal, constraints, non-goals і критерії готовності.
  • Під час роботи: проси агента читати існуючий код і дотримуватися локальних patterns.
  • Після змін: дивись diff, запускай тести, перевіряй edge cases і user-facing behavior.
  • Для команди: домовся про політику privacy, code ownership, review rules і допустимі AI-інструменти.
