Seniour / Core

Як modularization впливає на iOS-проєкт?

Коротка відповідь

Модульна архітектура розбиває застосунок на незалежні частини з чіткими межами відповідальності. Модулі зменшують coupling, дають ownership boundaries, пришвидшують incremental builds і дозволяють незалежне тестування. Але надмірна модульність створює складний dependency graph, boilerplate і overhead у release management.

Для iOS це зазвичай означає feature modules, shared/core modules, design system, networking, analytics, dependency direction, access control, Swift Package Manager, binary frameworks і build caching.

Навчальний розбір

Що треба зрозуміти

Модуль - це не просто папка. Це boundary: що модуль експортує назовні, що приховує всередині, від яких модулів залежить і хто може його змінювати. Добра модульність робить код зрозумілішим, а погана лише переносить хаос у багато маленьких targets.

Як пояснювати на співбесіді

Пояснюй через trade-offs: build time, testability, dependency direction, team ownership, API stability, visibility і migration cost. Senior-відповідь має показати, що ти не ділиш застосунок на модулі заради модулів, а вирішуєш конкретні проблеми продукту й команди.

Типові помилки

  • Створювати модуль на кожну дрібну сутність без реальної межі відповідальності.
  • Дозволяти feature-to-feature imports без правил dependency direction.
  • Експортувати всі типи як public, через що внутрішня реалізація стає контрактом.
  • Тримати shared module як смітник для всього, що “може знадобитись”.

Практика

Намалюй dependency graph для App, Core, DesignSystem і двох features. Потім спробуй пояснити, який модуль можна тестувати окремо, який має public API, а який повинен лишитись implementation detail.

Приклад структури модулів

App
├── FeatureProfile
│   ├── depends on: ProfileDomain, DesignSystem
│   └── exposes: ProfileFlowFactory
├── ProfileDomain
│   ├── depends on: Foundation
│   └── exposes: Profile, ProfileRepository
├── Networking
│   ├── depends on: Foundation
│   └── exposes: HTTPClient
└── DesignSystem
    ├── depends on: SwiftUI/UIKit
    └── exposes: AppButton, AppColor, Typography

Важлива ідея: feature може залежати від domain і design system, але domain не повинен знати про UI. App-композиція збирає залежності разом і вирішує, яку реалізацію передати у feature.

Фасадний підхід: назовні видно тільки public facade

Фасадне програмування в модульній архітектурі означає, що клієнти модуля бачать маленький стабільний API, а конкретні сервіси, mappers, cache і networking-код залишаються internal. Це зменшує coupling: модуль може міняти реалізацію без зміни зовнішнього контракту. 

// Sources/ProfileFeature/ProfileFeatureFacade.swift
public protocol ProfileFeatureFacade {
    func makeProfileScreen(userID: String) -> UIViewController
}

public enum ProfileFeatureFactory {
    public static func make(
        apiClient: APIClient,
        analytics: AnalyticsTracking
    ) -> ProfileFeatureFacade {
        DefaultProfileFeatureFacade(
            apiClient: apiClient,
            analytics: analytics
        )
    }
}
// Sources/ProfileFeature/Internal/DefaultProfileFeatureFacade.swift
internal final class DefaultProfileFeatureFacade: ProfileFeatureFacade {
    private let apiClient: APIClient
    private let analytics: AnalyticsTracking

    internal init(apiClient: APIClient, analytics: AnalyticsTracking) {
        self.apiClient = apiClient
        self.analytics = analytics
    }

    public func makeProfileScreen(userID: String) -> UIViewController {
        let service = ProfileService(apiClient: apiClient)
        let viewModel = ProfileViewModel(
            userID: userID,
            service: service,
            analytics: analytics
        )
        return ProfileViewController(viewModel: viewModel)
    }
}
// Sources/ProfileFeature/Internal/ProfileService.swift
internal final class ProfileService {
    private let apiClient: APIClient

    internal init(apiClient: APIClient) {
        self.apiClient = apiClient
    }

    internal func loadProfile(userID: String) async throws -> Profile {
        let dto: ProfileDTO = try await apiClient.get("/users/\(userID)")
        return ProfileMapper.map(dto)
    }
}

Клієнт модуля не знає про ProfileService, ProfileDTO, mapper чи конкретний ViewModel. Він працює лише з ProfileFeatureFactory і ProfileFeatureFacade. Це хороший варіант, коли треба приховати складну feature-реалізацію і залишити стабільний integration point для App-модуля.

Правила для модульної архітектури

  • Public API має бути маленьким, стабільним і навмисним.
  • Internal implementation може змінюватись без міграції клієнтів модуля.
  • Feature modules не повинні імпортувати один одного напряму без окремого contract module.
  • App target має збирати композицію залежностей, а не містити business logic.
  • Shared modules мають містити справді shared contracts, а не випадковий utility-код.
