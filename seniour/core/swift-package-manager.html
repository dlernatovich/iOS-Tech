Як працювати зі Swift Package Manager в iOS-проєкті?

Коротка відповідь

Swift Package Manager, або SPM, - стандартний інструмент Apple для опису Swift-пакетів, модулів, targets, products, dependencies і test targets. В iOS-проєктах його використовують для зовнішніх бібліотек, внутрішньої модульної архітектури, shared domain-коду, design system, networking layer і feature modules.

На співбесіді важливо пояснити не тільки як додати package, а й як контролювати versions, dependency graph, access control, public API, binary size, build time і ownership модулів.

Навчальний розбір

Що треба зрозуміти

Package описує, які modules збираються, які products доступні іншим пакетам і від яких dependencies вони залежать. Target - це одиниця компіляції. Product - це те, що інші можуть підключити: library, executable або plugin.

Як пояснювати на співбесіді

Почни з простого: SPM читає Package.swift, резолвить versions, будує dependency graph і компілює targets. Потім переходь до практики: pinning versions, package resolution, локальні packages, public/internal API, test targets і modular architecture.

Типові помилки

  • Підключати dependency до всіх targets замість конкретного target, якому вона потрібна.
  • Відкривати весь код через public, бо інакше “не видно з App”.
  • Не фіксувати версії й отримувати непередбачувані оновлення.
  • Змішувати feature code, shared code і app composition в одному package target.

Практика

Створи локальний package з двома targets: ProfileDomain і ProfileFeature. Зроби так, щоб App бачив тільки factory/facade, а services, DTO і mappers залишались internal.

Як додати package в Xcode

  1. Відкрий проєкт або workspace в Xcode.
  2. Обери File > Add Package Dependencies....
  3. Встав URL репозиторію, наприклад GitHub URL.
  4. Обери version rule: exact, branch, revision або semantic range.
  5. Вибери product і target, до якого його треба підключити.

Для production зазвичай краще використовувати semantic versions або exact versions для критичних залежностей. Branch dependency зручна тимчасово, але ризикована для стабільних релізів.

Як додати локальний package

Локальний package зручно використовувати для внутрішніх модулів: feature, domain, networking, design system. Його можна тримати в репозиторії поруч із app target. 

MyApp/
├── MyApp.xcodeproj
├── Packages/
│   └── AppModules/
│       ├── Package.swift
│       ├── Sources/
│       └── Tests/
└── MyApp/

В Xcode обери File > Add Packages... або просто перетягни локальну package-папку в workspace. Після цього підключи потрібні products до app target.

Приклад Package.swift для модульної архітектури

// swift-tools-version: 5.10
import PackageDescription

let package = Package(
    name: "AppModules",
    platforms: [
        .iOS(.v16)
    ],
    products: [
        .library(name: "ProfileDomain", targets: ["ProfileDomain"]),
        .library(name: "ProfileFeature", targets: ["ProfileFeature"]),
        .library(name: "Networking", targets: ["Networking"])
    ],
    dependencies: [
        .package(
            url: "https://github.com/apple/swift-collections.git",
            from: "1.1.0"
        )
    ],
    targets: [
        .target(
            name: "Networking",
            dependencies: []
        ),
        .target(
            name: "ProfileDomain",
            dependencies: []
        ),
        .target(
            name: "ProfileFeature",
            dependencies: [
                "ProfileDomain",
                "Networking",
                .product(name: "Collections", package: "swift-collections")
            ]
        ),
        .testTarget(
            name: "ProfileFeatureTests",
            dependencies: ["ProfileFeature"]
        )
    ]
)

Тут ProfileFeature залежить від domain і networking, але domain не знає про UI або network. Це допомагає тримати dependency direction чистим.

Як менеджити SPM-залежності

  • Оновлюй dependencies свідомо, бажано окремим PR із changelog і тестами.
  • Перевіряй Package.resolved: він фіксує конкретні revisions.
  • Не додавай dependency на рівень App, якщо вона потрібна лише одному feature target.
  • Слідкуй за transitive dependencies, binary size і license requirements.
  • Винось contracts у маленькі targets, якщо треба розірвати циклічні залежності.

Фасад у SPM-модулі

У Swift Package всі файли одного target бачать internal типи один одного. Інші modules бачать лише public або open. Це природно підходить для фасадного API. 

// Public API: видно з App target
public protocol PaymentsFacade {
    func startCheckout(orderID: String) -> UIViewController
}

public enum PaymentsModule {
    public static func make(api: APIClient) -> PaymentsFacade {
        DefaultPaymentsFacade(api: api)
    }
}
// Internal implementation: видно тільки всередині Payments target
internal final class DefaultPaymentsFacade: PaymentsFacade {
    private let api: APIClient

    internal init(api: APIClient) {
        self.api = api
    }

    public func startCheckout(orderID: String) -> UIViewController {
        let service = PaymentsService(api: api)
        let viewModel = CheckoutViewModel(orderID: orderID, service: service)
        return CheckoutViewController(viewModel: viewModel)
    }
}

internal final class PaymentsService {
    private let api: APIClient

    internal init(api: APIClient) {
        self.api = api
    }
}

App target імпортує PaymentsFeature і працює з PaymentsModule.make(...). Він не може напряму створити PaymentsService, бо це implementation detail.

Коли робити окремий module

  • Є чіткий owner або команда, яка відповідає за частину продукту.
  • Код має стабільний API і часто перевикористовується.
  • Потрібне ізольоване тестування без запуску всього app target.
  • Є проблеми з build time або занадто великим dependency graph.
  • Потрібно приховати складну implementation behind facade.
